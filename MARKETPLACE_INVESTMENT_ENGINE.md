# Marketplace Investment Engine — V1.2.2 (Master Specification)

**Status:** Architecture only. No implementation code, no UI, no APIs, no DB
migrations, no blockchain code.
**Locked references:** `DOMAIN_MODEL.md`, `ENTITY_RELATIONSHIP.md`,
`MIGRATION_STRATEGY.md`, `RELCKO_ECOSYSTEM_ARCHITECTURE.md`,
`MODULE_DEPENDENCY_MAP.md`, `EVENT_ARCHITECTURE.md`, `PERMISSION_MODEL.md`,
`IMPLEMENTATION_ROADMAP.md`.

This engine is the **origin of every economic event** in Relcko: investments,
resales, NFT ownership, governance power, dividends, treasury movement, and
referral commission all derive from it. It is the single source of truth for
property ownership and capital movement.

Companion specs (this milestone):
- `INVESTMENT_STATE_MACHINE.md` — investment lifecycle states + transitions.
- `PROPERTY_STATE_MACHINE.md` — property lifecycle states + transitions.
- `OWNERSHIP_MODEL.md` — fraction/NFT/SPV/corporate/joint/institutional ownership.
- `PAYMENT_SETTLEMENT_ARCHITECTURE.md` — payment methods, FX, settlement, refunds.
- `VALUATION_ENGINE.md` — valuations, NAV, ROI/Yield/IRR, cashflow.
- `DOCUMENT_MANAGEMENT_ARCHITECTURE.md` — document engine + compliance artifacts.
- `MARKETPLACE_EVENT_EXTENSION.md` — marketplace-specific event catalog.

## 1. Design principles (locked)

1. **Single source of truth for ownership.** `Ownership` (off-chain ledger) is
   authoritative; on-chain token balances are the verifiable mirror, reconciled
   continuously (see `OWNERSHIP_MODEL.md` §Reconciliation).
2. **Full traceability.** Every investment links Reservation → Payment →
   Compliance → Ownership → Commission → Treasury → Ledger via immutable ids.
3. **Auditability.** Every ownership change appends `AuditLog` (entity 19);
   every financial movement appends `Transaction` (entity 9).
4. **Immutable ledger entries.** `Transaction` is append-only; corrections are
   offsetting entries, never edits.
5. **Scale to millions.** Event-sourced core, CQRS read models, horizontal
   scaling (see §10).

---

## 2. Position in the ecosystem

The engine is a **shared platform capability**, not a standalone app module in the
UI sense. It is consumed by:
- **Marketplace module** (UI for browse/invest/resale — V1.2.0 browsing exists,
  invest flow specified here).
- Every downstream module via events (Portfolio, Treasury, Governance, Dividend,
  NFT, Network Engine, Referral Campaigns, AI Copilot, Document Vault, Global Map,
  Admin) — see `MODULE_DEPENDENCY_MAP.md`.

It depends only on the **shared platform layer** (Identity/Wallet, Ledger,
Property core, Money, Event Bus, Permission Service) exactly as
`RELCKO_ECOSYSTEM_ARCHITECTURE.md` §1 prescribes. It does **not** import other
modules.

## 4. Investment lifecycle (19 stages)

Mapped to states/events (full transitions in `INVESTMENT_STATE_MACHINE.md`):

```
Property Published ──▶ Investment Started ──▶ Reservation ──▶ Payment Processing
   │                                                                        │
   │                                                   KYC Verification ──▶ Compliance Review
   │                                                                        │
   └──▶ Investment Approved ──▶ Ownership Allocation ──▶ Fraction Mint Ready
                                     │
                                     ├─▶ Portfolio Updated
                                     ├─▶ Commission Generated
                                     ├─▶ Treasury Settlement
                                     ├─▶ Dividend Eligibility
                                     ├─▶ Governance Eligibility
                                     └─▶ Investment Complete

 Side branches: Cancelled · Refunded · Failed · Expired
```

Each stage is an explicit `Investment.state` (or `Property.state`) transition
emitting one or more canonical/extended events.

## 5. Primary market mechanics

| Mechanic | Design |
|----------|--------|
| Open investment | public round, no allocation gating. |
| Allocation limits | per-investor min/max from `Property.min_investment` + round caps. |
| Minimum / Maximum investment | `min_investment` (entity 1) + `round.max_per_investor`. |
| Whitelist rounds | `Round.allowlist` of investor ids / jurisdiction. |
| Private / Public rounds | `Round.visibility` = private | public; sequenced by `Round.order`. |
| Oversubscription | when `sold_tokens` would exceed `total_tokens`; triggers waiting list + allocation strategy. |
| Waiting list | `WaitingList` entity (investor_id, round_id, rank) when round is full. |
| Automatic allocation | pro-rata / FIFO on oversubscription per `Round.allocationStrategy`. |
| Manual allocation | Compliance/Property Manager allocation (audit-logged). |
| Soft cap / Hard cap | `Round.soft_cap`, `Round.hard_cap` (tokens or currency). |
| Campaign duration | `Round.starts_at`, `Round.ends_at`; auto-close on expiry → `ReservationExpired`. |
| Funding completion | `PropertyFunded` when `sold_tokens == total_tokens` (or hard cap met). |

## 6. Secondary market mechanics

| Mechanic | Design |
|----------|--------|
| List for sale | `MarketplaceListing` (entity 5) from `Ownership` delta. |
| Partial / Full sale | list a fraction `qty <= owned`. |
| Market order | immediate fill against existing offer. |
| Limit order | `MarketplaceListing` with `price` waiting for taker. |
| Offer system | counter-offers on a listing (`Offer` entity). |
| Negotiation | offer/counter-offer thread (audit-logged). |
| Escrow | `EscrowLedger` holds funds between agree and `SettlementCompleted`. |
| Settlement | atomic: `OwnershipTransferred` + `Payment` settled + `Commission` (secondary rate). |
| Ownership transfer | updates `Ownership` (source decreases, buyer increases). |
| Commission calculation | `Commission` generated on both primary and secondary (legacy gap fixed: webhook + admin paths). |

## 7. Financial engine — eight ledgers

All are **categorizations of the single immutable `Transaction` store** via a
`ledger` discriminator, with dedicated read models for reporting:

| Ledger | Category key | Purpose |
|--------|--------------|---------|
| Investment Ledger | `investment` | per-investment capital in/out. |
| Capital Ledger | `capital` | aggregate property capital raised. |
| Escrow Ledger | `escrow` | funds held pending settlement. |
| Settlement Ledger | `settlement` | completed payer→payee movements. |
| Commission Ledger | `commission` | agent/referrer earnings. |
| Treasury Ledger | `treasury` | platform/SPV custody movements. |
| Dividend Ledger | `dividend` | distributions to holders. |
| Tax Ledger | `tax` | withholding/issuance of tax documents. |

Invariant (from `ENTITY_RELATIONSHIP.md` §4): for every property,
`Investment Ledger in == Capital Ledger + Escrow + Settlement fees + Commission`
(modulo refunds). Reconciliation job validates this continuously.

## 10. Scalability (targets: 10M investors, 100K properties, millions of records)

- **Event sourcing:** `Investment`, `Ownership`, `Payment` are event-sourced;
  current state = fold of events; full rebuild possible from the log.
- **CQRS read models:** Portfolio, VotingPower, Leaderboard, Region stats are
  async projections (idempotent, keyed by aggregate id).
- **Sharding:** partition by `property_id` (property-centric) and `investor_id`
  (investor-centric); both derive from the same event stream.
- **Async workers:** payment processing, compliance review, mint, settlement,
  commission, dividend snapshot — all queued, retried, idempotent.
- **Caching:** property catalog cached at edge; ownership balances cached with
  short TTL + event-driven invalidation.
- **Horizontal scaling:** stateless API/worker tiers; ledger + projection stores
  horizontally partitioned; bus partitioned by aggregate.
- **Reconciliation:** continuous chain↔ledger and ledger↔ledger checks; alerts on
  divergence (`ComplianceFlagRaised`).
